# Working Backwards

Working Backwards is Amazon's product development method. Before building anything, the team writes a press release and an FAQ from the customer's point of view. If the value is not clear on paper, the product should not be built.

## Start With the Ending

Most teams begin with features and end with confusion.

Working Backwards flips the order. You imagine the product has already launched, then write the story customers would tell about it. If that story is not compelling, the core value is not ready.

The method forces value validation before code is written.

## The Two Documents

Working Backwards produces only two core documents:

1. A press release that explains the product in plain language
2. An FAQ that challenges the idea from every angle

These documents are not archive material. They are debate tools.

## PR Template

The press release should answer:

- What is the product?
- Who is it for?
- What problem does it solve?
- Why does it matter now?

## FAQ Template

The FAQ should answer questions like:

- What exactly is it?
- Why is it needed?
- How much does it cost?
- When does it launch?
- Who else is in the market?

## When to Use It

Working Backwards is best for:

- New product planning
- Major feature validation
- Aligning a team around a shared idea
- Communicating with executives or investors

## Kindle Case Study

Amazon used Working Backwards before launching Kindle.

The team wrote the story first, then used it to test whether the product was worth building. The launch narrative focused on fast wireless book downloads, a screen built for reading, long battery life, and access to a large book catalog.

## Five Principles

1. Write the press release before the FAQ.
2. Ask the most skeptical people to review it.
3. Expect the first draft to be rough.
4. Keep iterating until the team agrees the idea is worth pursuing.
5. Reuse the PR as a communication tool once the product exists.
