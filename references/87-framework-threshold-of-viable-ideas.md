# Threshold of Viable Ideas

The Threshold of Viable Ideas is a framework for deciding when an idea is good enough to pursue. Innovation should happen in a constrained environment, not in a zero-barrier free-for-all.

## Why More Ideas Can Be a Warning Sign

Too many ideas can create decision paralysis. When every idea feels equally exciting, the team loses the ability to tell which ones deserve time, money, and attention.

## The Six Criteria

| Criterion | Questions to Ask |
|---|---|
| Technical Feasibility | Can we actually build it? Do we have the skills and cost structure for it? |
| Business Viability | Can it make money or create strategic value? Is the market large enough? |
| User Desirability | Do users actually want it? What pain does it solve? |
| Time Feasibility | Can we build it before the opportunity closes? |
| Resource Availability | Do we have enough people, budget, and support? |
| Legal / Compliance | Does it meet legal, regulatory, and privacy requirements? |

## When to Set a Threshold

- When the project has a clear goal
- When resources are limited
- When time is tight
- When the team is exploring a new technical direction

## Google X Case Study

Google X, now X Development, is famous for filtering ideas very aggressively.

- Focus on moonshots with large upside
- Kill ideas quickly if the evidence is weak
- Use experiments and data instead of intuition alone

## How to Build a Threshold System

1. Choose the evaluation dimensions that matter to your business.
2. Define a scoring system for each dimension.
3. Set up a review process for new ideas.
4. Record decisions and revisit them later.

## Common Mistakes

| Mistake | Why It Hurts |
|---|---|
| Too many or too few criteria | Analysis becomes either noisy or meaningless |
| Ignoring hidden costs | Opportunity cost and switching cost matter |
| Using the same threshold for every project | Different ideas need different filters |
| Making the threshold static | Thresholds should change with the market |
| Over-relying on numbers | Some value is hard to quantify directly |

## History

| Period | Development |
|---|---|
| 1950s-1960s | NASA's Apollo decision discipline shapes early systems thinking |
| 1970s-1980s | Venture capital culture encourages faster filtering |
| 1990s-2000s | Agile and MVP thinking push validation earlier |
| 2010s to present | Design thinking and dual-track development make thresholds more practical |
