# Five-User Usability Testing

Jakob Nielsen's research showed that usability testing with a small number of users can reveal about 85% of usability problems. The reason is diminishing returns: each new user tends to reveal fewer new issues than the last.

## Why Five Users Are Often Enough

You do not need a huge sample to start finding real usability problems. The first few users tend to expose the biggest issues quickly.

- The first user may expose around 63% of the problems
- By the fifth user, you can often uncover 80% to 90% of the issues

The point is not statistical perfection. The point is to learn fast enough to improve the product.

## Issue Severity

| Severity | Meaning |
|---|---|
| Critical | The user cannot complete the task |
| Serious | The task becomes difficult or requires a workaround |
| Minor | The issue hurts the experience but does not block the task |
| Cosmetic | The issue affects appearance more than function |

## When to Use It

Five-user testing works best when you need fast feedback inside a short iteration cycle:

- Pre-launch usability checks
- Concept validation with low-fidelity prototypes
- Testing with first-time users
- Iterative refinement after release
- Cross-device or cross-platform checks
- Early accessibility screening

## Intuit TurboTax

Intuit used repeated small-sample testing cycles to improve TurboTax over time.

| Time | Phase | What Happens |
|---|---|---|
| January | Planning | Align product, design, engineering, data, and QA on goals and assumptions |
| February-March | Concept validation | Test low-fidelity prototypes and rough workflows |
| April-May | Round 1 | Test the most important functions with about five representative users |
| June-July | Fixes | Improve the product and plan the next round |
| August-September | Round 2 | Verify that the fixes actually work |
| October-December | Launch and review | Ship, then test again to confirm improvement opportunities |

## Five Keys

1. Use different users in each round when possible.
2. Do not test only with internal staff or familiar users.
3. Let people think aloud while interfering as little as possible.
4. Prioritize issues by severity.
5. Observe real behavior and ask why they acted that way.

## History

| Era | Development |
|---|---|
| 1980s | Large organizations begin formal usability work |
| 1990s | Nielsen publishes the five-user rule |
| 1990s | Diminishing returns in issue discovery are quantified |
| 2000s | The method becomes common in product teams |
| 2010s-present | Five-user testing is integrated into agile workflows |
