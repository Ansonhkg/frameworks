# Wizard of Oz Testing

Wizard of Oz testing lets you validate whether users will use a feature before the technology behind it exists. The user sees a working product; the system response is actually simulated by a human behind the scenes.

## Why It Exists

Early product teams often need to answer two questions before investing in full development:

- Do users really want this feature?
- What do they expect it to do?

Wizard of Oz testing solves this by making the interface feel real while the implementation remains manual.

## Front End vs Back End

The user sees a functioning product. The truth is that no real automation is happening yet.

| User sees | Reality |
|---|---|
| A live-looking interface | No real system logic running |
| Input fields and buttons | Human operators simulate responses |
| Fast feedback | Manual responses delivered in real time |
| An intelligent assistant | A person is operating behind the curtain |

## How the Test Works

1. The user submits a request, such as "Write me ad copy."
2. The interface shows a processing state.
3. A researcher or product team member reads the request.
4. The operator manually writes the response.
5. The user believes they are using a real intelligent system.

The key is that the user behaves as if the system is real, which gives you honest behavioral data.

## When to Use It

Wizard of Oz testing is useful when:

- You want to validate demand before building a complex feature
- You are exploring AI or assistant experiences
- You are testing a new interaction model
- You want to learn cheaply before writing code

## IBM Speech Recognition Research

The method traces back to IBM research in 1984, when John F. Kelley and his team studied how people reacted to speech interfaces before the technology was ready.

They simulated the system manually and learned that users were willing to treat the experience as if it were real. That insight influenced later UX and product research work, especially in voice and assistant interfaces.

## Five Keys

1. Preserve realism so users do not realize the system is manual.
2. Make the operator behave like the system.
3. Observe behavior, not just what users say.
4. Keep the manual simulation consistent.
5. Debrief users after the test.

## Why It Matters

Wizard of Oz testing lets you validate the value of an idea before you build the technology. That saves money and reduces the risk of building the wrong thing.
