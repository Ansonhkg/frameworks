# Feature Priority Pyramid

The Feature Priority Pyramid helps product teams decide what to build when resources are limited. It places the most critical, foundational work at the bottom and more advanced or experimental work at the top.

## Why It Matters

Teams often rush to build shiny features while basic reliability is still weak. The pyramid keeps priorities in order so the product can stay stable, useful, and eventually differentiated.

## Five Layers

| Layer | Focus | What It Means |
|---|---|---|
| 1 | Stability | Fix crashes, broken flows, and critical bugs first |
| 2 | Core usability | Make the main user journey work smoothly |
| 3 | Feature completeness | Cover the essential needs of the product |
| 4 | Differentiation | Improve the parts that create clear advantage |
| 5 | Innovation | Explore new ideas once the foundation is solid |

## When to Use It

- When the team is arguing about what to do first
- When the backlog is too large
- When different departments disagree on priority
- When leadership needs a clear explanation for delays

## Notion Case Study

Notion is a good example of this thinking:

- Early stage: focus on crash fixes, database performance, and sync reliability
- Growth stage: improve interaction design, templates, and collaboration
- Mature stage: add AI, automation, and broader workflow support

The message is simple: do not build the top of the pyramid before the base is stable.

## How to Run a Priority Review

1. Collect every request in one list.
2. Place each request into one pyramid layer.
3. Check whether the base layers are underfunded.
4. Check whether higher layers are receiving too much attention.
5. Rebalance resources accordingly.

## Common Misuses

| Mistake | Why It Fails |
|---|---|
| Skipping the base layer | New features will fail if the product is unstable |
| Marking everything urgent | Nothing is truly prioritized |
| Ignoring UX quality | Working features can still feel bad to use |
| Innovating without a goal | Innovation should support the business, not distract from it |

## Practical Advice

- Review the pyramid every quarter
- Include product, design, engineering, and support in the discussion
- Use data to validate priority decisions
- Treat the pyramid as guidance, not as a rigid rulebook
